# Quality Pass Policy (Global)

이 문서는 OAC 기반 워크플로우에서 기능 구현 이후 수행하는 Quality Pass(메모리/자원/성능) 규칙을 정의한다.

## 1) 기능 subtask ↔ Quality Pass subtask (1:1)
- 모든 기능 subtask마다 반드시 대응되는 Quality Pass subtask를 1:1로 붙인다.
- 기능 구현만 완료된 상태로 상위 task를 done 처리하지 않는다.

## 2) Quality Pass Done 기준
- fast-check(테스트/빌드) 통과
- 최소 반복 실행(soak)에서 메모리/핸들/리소스가 지속 증가하지 않음
- 의심 징후가 있으면, 수정 전에 반드시 검출용 리그레션(테스트/스크립트)을 먼저 추가

## 3) 재개발/리팩터 트리거
- 동일 결함이 2~3회 반복될 때
- 땜질 수정이 여러 위치로 확산될 때
- 전역 수명(lifecycle) 또는 자원 소유권 문제가 의심될 때
- 위 조건 중 하나라도 충족하면 refactor subtask를 생성해 최소 단위로 재개발한다.

## 4) 성능/자원 점검 단계화
- 기본 순서는 "라이트 체크 → 필요 시 프로파일링"이다.
- 고비용 프로파일링은 기본 단계 통과 후, 증거 기반으로만 수행한다.
